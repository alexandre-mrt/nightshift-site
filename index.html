01 Architecture

Night Shift v5 unifies the best patterns from four previous versions into a single autonomous system. Every design decision optimizes for one thing: reliable code output while the developer sleeps.

Core Insight

Most agent harnesses are while loops with a model call inside. Night Shift is a state machine where every failure mode has a specific recovery path. The difference: an agent that crashes at 3am vs one that backs off, retries with a different strategy, and keeps working.

State persistence

JSON with compact fields

STATE.json uses short field names (t, s, p, grp) saving ~25% tokens per read. Auto-trimmed to <50 lines. jq for selective reads.

Inter-agent protocol

XML-tagged compressed NL

Telegraphic English in XML boundaries. No Markdown formatting. Abbreviations (fn, dep, impl, cfg). AI-to-AI only, PROBLEMS.md for humans.

Context management

KV-cache optimized

Stable prompt prefix (role, rules, format), variable suffix (task, files, lessons). Same template skeleton across agents maximizes cache hits.

02 The Loop

Each iteration starts with a fresh context window. State survives in files. The ralph-loop stop hook intercepts every exit and re-feeds the prompt.

1. Read STATE.json-> 2. Read ERRORS-> 3. TODO.md anchor-> 4. Read prefetch-> 5. init.sh-> 6. git log-> 7. Doom-loop check-> 8. Idle detection-> 9. Test-first check-> 10. Spawn agents-> 11. Pipeline prefetch-> 12. Early abort triage-> 13. Merge + validate-> 14. Error escalation-> 15. Critic (GAN)-> 16. QA cycle-> 17. Update state-> 18. WIP commit-> 19. Done? Gate : Loop

Pre-flight (Step 0)

Before the loop starts, Night Shift runs a thorough pre-flight: domain research via web search and Context7, a questionnaire to the user (who is about to sleep), ground truth document generation, test spec writing, and plan approval. Every ambiguity left unresolved becomes a hallucination at 3am.

Pipelining (Step 11) NEW in v5

While agents run in background, the orchestrator prefetches context for the next iteration: reads files for the next batch of tasks, pre-computes merge order, stores results in STATE.json. Next iteration skips exploration and goes straight to execution.

Test-First Flow (Step 9)

Every task starts with tests written first (red phase). The night-coder implements until tests pass (green). This gives each iteration an objective convergence criterion instead of subjective "looks done".

03 Heuristics & Self-Healing

Night Shift runs while you sleep. It needs to detect when it is stuck and recover without human intervention. These heuristics are the immune system.

Doom-Loop Detection

Runs every iteration. Catches circular behavior before it burns the budget.

SignalThresholdResponse
Same file edited3+ timesRevert to checkpoint, different approach
Same error hash3+ timesUpgrade model tier (haiku -> sonnet -> opus)
Same test failing3+ timesRe-examine the test itself (might be wrong)
No completions5+ iterationsForced mid-course re-evaluation
Convergence rate< 0.1Strategy change required
Diff hash reappearsA->B->ARevert to earliest clean state, different architecture

3-Tier Error Escalation

Tier 1 (attempts 1-3)

Normal retry with Reflexion

night-fixer (haiku) with self-critique before each attempt. Prompt variation to avoid pattern lock-in. Each retry reformulated slightly.

Tier 2 (attempts 4-6)

Rollback + Agent Team

git checkout to last clean commit. 3-5 agent team members investigate different hypotheses, challenge each other. Fundamentally different architecture required.

Tier 3 (7+)

BLOCKED

Mark task blocked. Log full trajectory to ERRORS.json + PROBLEMS.md. Commit WIP, move to next independent task. The user reviews in the morning.

Reflexion Pattern

Before each fix attempt, the agent writes a self-critique: "My previous approach failed because X. The root cause is likely Y. I will try Z instead." This gets logged to ERRORS.json, preventing the same mistake from repeating across iterations.

Prompt Variation on Retry

Identical prompts trigger identical reasoning patterns. On retry, each prompt is reformulated: reordered constraints, different framing, altered vocabulary for the same intent. This breaks pattern lock-in where the model keeps producing the same failed output.

Early Abort on Critical Failure NEW in v5

If any agent in a parallel batch returns a critical failure (build broken, type cascade), remaining agents are NOT merged. The critical error is fixed first, then valid results are selectively merged. Inspired by Claude Code's siblingAbortController pattern.

Anti-Hallucination Protocol

The user is asleep. Never guess APIs, types, or business logic. Blocked is better than broken.

Every agent follows five rules: investigate before coding, never guess (search first), never leave silent traces (no TODO stubs without PROBLEMS.md entry), log uncertainties with greppable markers (NIGHT-SHIFT-REVIEW), and choose blocked over broken.

Each agent type has a specific role, model tier, and isolation level. The orchestrator stays lean: it reads state, spawns agents, merges results, validates.

Safety Tiers

Schema-level tool filtering: reviewers never write, QA never edits, research never executes. This is enforced at the agent definition level, not by trust.

Model Upgrade Rule

If a sonnet agent fails 2x on the same task, it automatically retries with opus. The upgrade is logged in ERRORS.json for cost tracking.

Cache-Optimized Prompt Template NEW in v5

Stable prefix (cached)

Agent role, anti-hallucination rules, output format, concurrency rules, build/test/lint commands. Identical across all agents of the same type. Stays in KV-cache.

Variable suffix (per-task)

Task description, file list, interfaces to respect, lessons from ERRORS.json. Changes every spawn. Only this part invalidates cache.

Concurrency Classification NEW in v5

Every agent prompt includes concurrency rules: Read/Grep/Glob run in parallel (safe), Write/Edit run sequentially (never two on same file), Bash(git) runs sequentially, Bash(build/test) runs after all writes complete. Prevents race conditions in parallel tool execution.

05 Stability Gate & Audit

Before the PR is created, code passes through a multi-layer quality gate. The gate runs inside a single wrapper agent to protect the orchestrator's context.

GAN Pattern — Critic Review

After each merge, a critic sees ONLY the spec and the output code. It has zero knowledge of the generator's intent, reasoning, or conversation history. This information isolation prevents confirmation bias. The critic judges the artifact cold, as a new developer encountering it for the first time.

Audit Wrapper Agent

The entire stability gate runs inside ONE wrapper agent. The orchestrator reads only the ~20-line result file instead of accumulating 50+ agent reports. This is the key context-efficiency innovation from v3.5.

6 Review Personas

Persona 1

Architect

System design, module boundaries, dependency direction. Validates against GROUND_TRUTH_BRAINSTORM.md.

Persona 2

Domain Expert

Business logic correctness, edge cases from spec, data flow validation.

Persona 3

Code Expert

DRY, type safety, resource leaks, dead code, naming conventions.

Persona 4

Performance Expert

Unnecessary allocations, N+1 queries, missing indexes, bundle size.

Persona 5

Security Expert

OWASP top 10, secrets, injection, auth, input validation.

Persona 6

Human Advocate

Usability, accessibility, error message clarity, mobile responsiveness.

Cross-Validation Matrix

Personas are expected to overlap and challenge each other: Architect + Code on module boundaries, Security + Performance on rate limiting, Domain + Human on error messages, Performance + Security on caching (cache poisoning?).

Gate Protocol

Run 6 personas-> Collect findings-> 0 critical/high?-> clean++-> 3 consecutive?-> PASS

If any critical/high findings: fix with night-fixer, reset clean counter to 0, re-run. If >70% findings repeat from previous loop: exit (criticism loop detected). Safety cap: 10 loops max, then WIP PR.

Functional Verification Gate

Build + lint green is necessary but not sufficient. One real transaction on a local validator would have caught four bugs that shipped in v2.

Before the PR is created: start the application, run happy path end-to-end with real data, test 3+ failure scenarios. Backend: real HTTP requests. Smart contracts: deploy + real transactions. Frontend: navigate all pages, fill forms.

06 Context & Cost Management

The orchestrator's context window is the scarcest resource. Every design decision in v5 protects it.

Context Budget Rule

Before spawning agents: if this iteration already processed 5+ agent results OR read 3+ large files (>200 lines), commit state and exit. Ralph-loop gives fresh context next iteration. Hard rule: after merging results, discard raw agent output, keep only the <result> block.

Auto-Trimming

FieldRule
log arrayKeep last 5 entries only. Git log is the archive.
Completed tasksStrip tries/blocked fields
doom.file_editsReset counters for resolved items
gate.prev_hashesKeep only last loop's hashes
prefetchOverwrite each iteration (never accumulate)

State Format Decision

Research across TOON, YAML, JSON, XML, and binary formats (MessagePack). Result: JSON with compact field names is optimal. jq is essential for selective reads. Compact names (t, s, p, grp instead of title, status, priority, parallel_group) save ~25% tokens. XML-tagged compressed natural language for inter-agent protocol (minimal tag overhead, good semantic boundaries).

KV-Cache Optimization

Frozen prefix: system instructions, tool definitions, and skill content form the prompt prefix. Never remove or reorder tools mid-run. Append-only dynamic content at the END. Stable agent prompt template structure (vary only dynamic fields). This gives ~10x cost reduction on cached tokens across the session.

Cost Tracking

Budget enforcement

70% Switch non-critical agents to haiku
100% Graceful shutdown, WIP PR

Session metrics NEW in v5

Every session appends to ~/.claude/metrics/sessions.jsonl: date, project, iterations, tasks completed/blocked, duration, cost.

07 Ground Truth Documents

Before any code generation, three immutable reference documents are created. All agents validate against these, not the original user request. This prevents specification drift during long autonomous sessions.

Document 1

GROUND_TRUTH_BRAINSTORM.md

Explores all possible approaches with pros/cons. Orchestrator picks one. Alternatives stay for mid-course re-evaluation at 50% completion.

Document 2

GROUND_TRUTH_HEALTH.md

Audits current codebase: file structure, key modules, test coverage, known tech debt, fragile areas, dependency versions.

Document 3

GROUND_TRUTH_DOCS.md

Audits existing types, interfaces, API contracts, config files, env vars, naming/error handling/auth patterns.

These documents are IMMUTABLE after creation. If the agent discovers during execution that an approach from the brainstorm is better, it re-evaluates at the mid-course checkpoint, not ad-hoc.

08 Evolution

Night Shift v5 is the result of four iterations, each addressing failures discovered in production.

v5 (Unified) current
Best of v3.5 + v4. Cache-optimized prompts, pipelining, early abort, concurrency classification, compact state fields, metrics logging. Single skill replaces four variants.
v4 (Experimental)
Test-first flow (red->green). Ground truth documents. 6 audit personas via agent teams. Cost tracking with budget enforcement. Safety tiers for tool filtering.
v3.5 (Context-lean)
Compact AI-to-AI protocol (XML-tagged compressed NL). Audit wrapper agent (one agent for entire stability gate). Auto-trimming state. Context budget rule. TODO.md attention anchor.
v3 (Agent Teams)
Agent team debugging for Tier 2 escalation. Multiple review agents in audit loop. First attempt at structured inter-agent communication.
v2 (First Stable)
Ralph loop mechanism. Worktree isolation. Basic error escalation. Functional QA via Playwright. Learned the hard way: build+lint green is not sufficient.
